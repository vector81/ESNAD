# Esnad System Design

Production architecture for **Esnad Center for Studies and Research** — a bilingual (Arabic/English) academic publishing platform.

## System Overview

                    |  (CDN + SSR)    |

              +--------------+--------------+

     | sites/public     |          | sites/editor    |

## Architecture Decisions

### Dual-Frontend, Single Source

Two Vite builds (`vite.public.config.ts`, `vite.admin.config.ts`) compile from the same `src/` tree with different entry points. This gives:

- Independent deployment cadence per site
- Smaller bundles (public site excludes editor dependencies)
- Shared type safety across both apps

### Content Model

Single `publications` collection in Firestore. Each document carries:

| Field | Purpose |
|-------|---------|
| `slug` | URL identifier (may contain Arabic) |
| `type` | `article` or `book` — drives studio/reader behavior |
| `kind` | `book` or `research-paper` — drives routing (`/books/` vs `/library/`) |
| `status` | `draft` or `published` — public site filters on `published` |
| `workflow_stage` | `draft > in_review > needs_revision > approved > published` |
| `access_tier` | `free` or `paid` — gates PDF download behind Stripe checkout |
| `language_mode` | `ar`, `en`, or `both` — controls which localized fields are required |
| `content_json` | Tiptap ProseMirror JSON — the rich-text body |

23 publication categories covering studies, policy papers, reports, translations, infographics, etc.

### Slug Strategy

- **Stored slug**: Generated by `slugify()` — may contain Arabic Unicode (`\p{L}`)
- **Share slug**: Generated by `getShareSlug()` — always ASCII via `any-ascii` transliteration
- **Resolution**: `getPublicationBySlug()` matches on both stored slug and computed Latin slug
- **Why**: WhatsApp, Telegram, and X mangle Arabic characters in URLs. Share links must be Latin-only.

### API Layer (Serverless)

Vercel Functions in `api/`:

| Endpoint | Auth | Purpose |
|----------|------|---------|
| `article-shell.js` | None | SSR shell with OG meta for shared links |
| `checkout-session.js` | Firebase token | Creates Stripe checkout session |
| `confirm-purchase.js` | Firebase token | Grants access after payment |
| `contact.js` | None | Contact form submission via Resend |
| `subscribe.js` | None | Email newsletter signup |
| `subscribers.js` | Admin email | List subscribers (admin only) |

Shared utilities in `api/_lib/`: Firebase Admin SDK init, HTTP helpers (CORS, JSON, bearer token), Resend email rendering.

### Studio (Rich-Text Editor)

Full authoring environment at `src/studio/`:

- **TiptapEditor**: ProseMirror-based with custom nodes (Citation, Equation, Figure, Footnote) and marks (Comment)
- **Chapters**: Multi-chapter book support (`studio/lib/chapters.ts`)
- **Versioning**: Draft snapshots (`studio/lib/versions.ts`)
- **Review flow**: Comment threads with resolve/unresolve (`studio/lib/comments.ts`)
- **Workflow**: Stage transitions enforced by `canTransitionTo()` in `types/studio.ts`

### Reader

`src/reader/` renders published content:

- **ArticleReader**: Single-document view
- **BookReader**: Multi-chapter navigation
- **render-pm-json**: Custom ProseMirror JSON to React renderer (equations via KaTeX, citations, footnotes)

## Data Flow

### Publishing Flow
```
Author drafts in Studio
  -> saves content_json to Firestore
  -> workflow: draft -> in_review -> approved -> published
  -> public site reads published documents
  -> article-shell.js renders OG meta for social sharing
```

### Purchase Flow
```
User clicks Buy on PublicationPage
  -> createCheckoutSession() -> Stripe
  -> Stripe redirects to confirm-purchase endpoint
  -> grantPurchasedItem() writes to user's library
  -> PDF download unlocked
```

### Share Flow
```
User clicks Share on PublicationPage
  -> getShareSlug() transliterates title to ASCII
  -> builds URL: {SITE_URL}/{lang}/{section}/{latin-slug}
  -> copies to clipboard
  -> recipient opens link
  -> getPublicationBySlug() resolves Latin slug to publication
```

## External Services

| Service | Role | Config |
|---------|------|--------|
| **Firebase Firestore** | Document database | `VITE_FIREBASE_*` env vars |
| **Firebase Auth** | User authentication | Email/password + admin gate |
| **Cloudinary** | Image hosting/transform | `VITE_CLOUDINARY_*` env vars |
| **Stripe** | Payment processing | Server-side keys in Vercel env |
| **Resend** | Transactional email | API key in Vercel env |
| **Vercel** | Hosting + serverless | Two projects (public + editor) |

## Security Model

- **Public site**: Read-only, no auth required for browsing
- **Editor site**: Firebase Auth required, admin email allowlist in `subscribers.js`
- **API endpoints**: Bearer token verification via Firebase Admin SDK
- **Content sanitization**: DOMPurify for any rendered HTML
- **Firestore rules**: Server-side access control (separate from frontend)
